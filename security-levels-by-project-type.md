# プログラム種別ごとのセキュリティレベル指針

Tomoさんの C:\dev 配下のリポジトリを念頭に、「どの種類のプログラムに、どのレベルのセキュリティレビューを掛けるべきか」の見解をまとめたもの。
セキュリティレビュー実施時に、CCへの指示文の「観点」セクションを組み立てる素材として使う想定。

---

## 基本の考え方

セキュリティレベルは「コードの行数」や「技術の複雑さ」ではなく、次の3つで決まる。

1. **攻撃面(Attack Surface)** — 外部からの入力を受けるか。ネットワークに口を開けているか
2. **守るべき資産** — 認証情報、個人データ、金銭、実行権限など、漏れたり奪われたりすると困るものがあるか
3. **被害の波及範囲** — 侵害されたとき、そのプログラム単体で済むか、他のシステム(PC全体、他リポジトリ、口座)まで及ぶか

この3軸で見ると、同じ「Webサイト」でも要求レベルがまったく違うことが分かる。

---

## レベル分類(C:\dev のリポジトリを当てはめて)

### レベル1: 静的・クライアント完結型 — 「軽め」でよい

**該当:** spctrl-site / fisheye-tool / QED / ooi-quest

サーバーを持たず、GitHub Pagesなどで配信される静的サイト・単一HTMLゲーム。
守るべき秘密がそもそも存在せず、侵害されても被害は「そのページの閲覧者」に限定される。

**それでも見るべき観点:**

- **シークレットの誤コミット** — APIキーやトークンをうっかりコミット履歴に残していないか(履歴に一度入ると消すのが大変)
- **XSS** — URLパラメータやlocalStorageの値をinnerHTMLに流していないか。ゲームのセーブデータ読み込みも入力経路
- **外部CDN依存** — Three.jsなどをCDNから読む場合、バージョン固定とSRI(integrity属性)があるか。CDN側が侵害されると自サイトも道連れになる
- **依存パッケージ** — npm依存があるなら既知脆弱性チェック(`npm audit`)

**レビュー頻度の目安:** 大きな機能追加時、または半年に1回程度の棚卸しで十分。

---

### レベル2: 静的サイト + クライアント側認証 — 「中」だが特殊な注意点あり

**該当:** strata-site(admin-auth.js / bcryptjs)

ここが一番「誤解しやすい」ゾーン。重要な原則:

> **クライアント側認証は「鍵」ではなく「目隠し」である。**

bcryptハッシュをJSに埋め込む方式は、ハッシュ自体が公開物になる。つまり:

- 攻撃者はハッシュを持ち帰って**オフラインで総当たり**できる(レート制限が効かない)
- 認証の先にあるコンテンツも、結局は静的ファイルとして配信されているなら**URLを知れば直接取れる**

**見るべき観点:**

- パスワード強度の前提 — オフライン攻撃前提なら、辞書に載らない長いパスフレーズが必須(cost factor 10は妥当だが、パスワードが弱ければ無意味)
- 「認証の先」に置いてよいものの線引き — 本当に秘匿すべきデータをこの方式の裏に置かない。置くならサーバー側認証が必要、という判断基準を明文化しておく
- ハッシュ・ユーザー名のハードコード禁止原則の遵守(既に確立済みの原則のチェック)
- セッション管理 — 認証状態をどこに持つか(sessionStorage等)と、その改ざん耐性は「ない」前提で設計されているか

**レビュー頻度の目安:** 認証まわりを触るたび。

---

### レベル3: ローカルサーバー / API / MCP・承認基盤 — 「高」。最重点

**該当:** Vibeline(Express/SQLite + MCP) / subu-cp

ここが C:\dev の中で**最もセキュリティ投資すべき場所**。理由は明快で:

- ネットワークに口を開けている(:3000 / :3001、将来的にCloudflareトンネルで外部公開)
- **CCの実行を承認するゲートそのもの**なので、ここが破られると「攻撃者がCCに任意のタスクを承認できる」= PC上での任意コード実行に直結する
- 認証情報(bcrypt、.env)とAudit trailという守るべき資産を持つ

**見るべき観点:**

- **認証のfail-closed** — 認証設定が壊れている/未設定のとき、開くのではなく閉じるか(subu-cpで一度修正済みの観点。Vibelineにも同じ検査を)
- **パストラバーサル** — ファイルパスを受け取るAPIで `../` を弾いているか
- **インジェクション** — SQLite(better-sqlite3)はプリペアドステートメント徹底か。コマンド実行系は引数を組み立てていないか
- **シークレット管理** — .envがgitignoreされているか、CLI引数に認証情報を載せていないか(過去のインシデントで確立したCLAUDE.mdルールの遵守確認)
- **トンネル公開時の前提変化** — subuta.winで外に出した瞬間、「LAN内だから大丈夫」の前提が全部崩れる。公開前に必ず再レビュー
- **承認ログ(audit trail)の改ざん耐性** — ガバナンスが価値の製品なので、ログ自体が資産
- **CORS / オリジン検証** — Expoアプリ以外からAPIを叩かれる経路がないか

**レビュー頻度の目安:** リリース(SemVerマイナー)ごと + 外部公開などアーキテクチャ変更時は必須。

---

### レベル4: モバイルアプリ(クライアント) — 「中」

**該当:** Vibeline Expoアプリ

サーバーほどではないが、承認システムの「リモコン」なので:

- **トークン/認証情報の保存場所** — AsyncStorage平文ではなくexpo-secure-store(Keychain/Keystore)か
- APIエンドポイントと通信のHTTPS化(トンネル公開後)
- 承認ボタンの誤操作・なりすまし防止(端末を他人に触られたケースをどこまで想定するか)

---

### レベル5: 金銭が動くもの — 「最高」だが性質が違う

**該当:** FX_Guardian(MT4 EA、現在休止中)

行数は小さくても、バグ=即・金銭被害という意味で別格。
ただしMT4のサンドボックス内で動くため、Web系の脆弱性よりも:

- 口座認証情報の管理(VPS上の保管方法)
- ロット計算・損切りロジックの**フェイルセーフ**(セキュリティというより安全工学)
- VPS自体のOSセキュリティ(RDPの認証、Windows Update)

が論点になる。再開時には専用のチェックリストを作るのが良い。

## レビュー結果の置き場所について

**推奨: README.md ではなく専用ファイル。**

理由:

1. **READMEは「利用者向けの顔」** — プロジェクトの説明・使い方を書く場所。脆弱性の指摘リストが載っていると、利用者にとってノイズであり、公開リポジトリなら攻撃者へのヒント集にもなりかねない
2. **ライフサイクルが違う** — READMEは安定文書、レビュー結果は「指摘→修正→クローズ」と動く作業文書。混ぜると差分が追いにくい
3. **STATUS.mdパターンとの一貫性** — 「耐久性のある状態は専用の versioned ファイルに」という既存の運用と同じ思想

**具体的な提案:**

- `SECURITY-REVIEW.md`(リポジトリ直下) — レビュー結果・指摘・対応状況。CCに書かせるのはこれ
- 名前を `SECURITY.md` にしないのがポイント。GitHubでは SECURITY.md は「脆弱性の報告窓口を示すポリシーファイル」という公式の意味を持つ規約があり、リポジトリの Security タブに表示される。レビュー結果置き場と混ぜない方がよい
- READMEには必要なら1行だけ: 「セキュリティレビュー状況は SECURITY-REVIEW.md を参照」

**書式の提案(CC指示文に含める用):**

```markdown
# Security Review — {repo名}
最終レビュー: YYYY-MM-DD / 対象バージョン: vX.Y.Z / レビュー実施: CC (model名)

## サマリー
- 重大: n件 / 中: n件 / 低: n件

## 指摘事項
### [重大度] タイトル
- ファイル/行:
- 内容:
- 修正方針:
- 状態: 未対応 / 修正済み(vX.Y.Z) / 対応しない(理由)
```

「状態」列を持たせておくと、修正タスクをHITLで1件ずつ回すときの台帳としてそのまま使える。
